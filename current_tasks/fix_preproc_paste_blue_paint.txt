Task: Fix preprocessor ## token paste with blue-painted (self-referential) tokens

Problem:
When a macro name is passed as an argument through multiple macro layers and
eventually used with ## (token paste), the blue paint marker (0x01) from
self-referential expansion prevention corrupts the pasted token.

Example:
  #define __cmp_op_max >
  #define __cmp(op, x, y) ((x) __cmp_op_##op (y) ? (x) : (y))
  #define __cmp_once(op, x, y, ux, uy) ({ typeof(x) ux=(x); typeof(y) uy=(y); __cmp(op, ux, uy); })
  #define __careful_cmp(op, x, y) __builtin_choose_expr(..., __cmp(op, x, y), __cmp_once(op, x, y, __x, __y))
  #define max(x, y) __careful_cmp(max, x, y)

  max(a, b)  // FAILS: __cmp_op_max not expanded because blue paint on 'max' token

Root cause:
1. max(a,b) expands. 'max' is added to expanding set.
2. __careful_cmp prescans arg 'op' = "max" with expanding={"max"}, producing "\x01max"
3. This blue-painted token flows through __cmp_once to __cmp
4. In __cmp, ## pastes produce "__cmp_op_\x01max" instead of "__cmp_op_max"
5. The pasted token doesn't match macro __cmp_op_max

Impact:
This blocks 19/20 random kernel files (all use Linux min/max macros from minmax.h).

Fix:
Strip BLUE_PAINT_MARKER bytes from raw args when used in ## paste operations.
The C standard says ## creates a new token which is rescanned; blue paint from
operands should not carry through to the pasted result.
